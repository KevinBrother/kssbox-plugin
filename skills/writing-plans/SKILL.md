# writing-plans Skill

## 在流程链路中的位置

```
product-strategy → prd-writer → [writing-plans] → harness-sync → executing-plans
```

**输入**：PRD 文档（来自 prd-writer，或用户提供的任意需求描述）  
**输出**：`.plans/YYYY-MM-DD-<feature>.md` — 与 executing-plans 之间的解耦契约文件

---

## 前提检查

| 用户提供的内容 | 处理方式 |
|------------|--------|
| 完整 PRD 文档 | 直接读取，开始拆解 |
| 部分需求描述 | 自行提炼关键功能列表，告知用户补充了哪些假设 |
| 什么都没有 | 问 1 个问题：这次要开发什么功能，大概涉及哪些模块？ |

---

## 执行流程

### 步骤一：解析需求（串行，必须先完成）
→ 读取 PRD 或用户描述  
→ 提取：功能模块列表、核心数据流、外部依赖、验收标准  
→ 向用户确认功能边界和优先级，**等确认后**进入步骤二

### 步骤二：设计文件结构（依赖步骤一）
→ 读 `references/phase-2-file-structure.md`  
→ 列出每个文件的职责，明确新建/修改/删除  
→ 遵循原则：单一职责、按职能分组（而非按技术层）

### 步骤三：拆解任务（依赖步骤二）
→ 读 `references/phase-3-task-breakdown.md`  
→ 每个任务对应一个内聚功能单元  
→ 每个步骤 2-5 分钟可完成，测试先于实现（TDD）

### 步骤四：生成计划文档
→ 同时读取（可并行）：
  - `references/plan-template.md` — 输出格式模板
  - `references/quality-check.md` — 质量检查清单  
→ 按模板生成完整计划，逐条过检查清单

### 步骤五：落盘
→ 保存到 `.plans/YYYY-MM-DD-<feature>.md`（路径相对于目标项目根目录）  
→ 输出计划文件路径，告知用户：**下一步运行 executing-plans，指定此文件**  
→ 不落盘、不自检，不算完成

## 计划文档结构

```
# [功能名] 实现计划

> 本文件是 executing-plans 的输入。执行前不得修改任务步骤顺序。

**目标：** [一句话说清楚这个计划交付什么]
**PRD 来源：** [PRD 文件路径或描述]
**预估任务数：** N 个
---

## 文件结构

| 操作 | 路径 | 职责说明 |

---

## 任务列表

### Task N: [组件名]

> [这个任务交付什么，一句话]

**关联文件：** path/to/file.ts

- [ ] 写失败的测试
- [ ] 运行测试确认失败
- [ ] 实现最小可用代码
- [ ] 运行测试确认通过
- [ ] git commit -m "feat: ..."

---

## 验收标准

- [ ] 所有测试通过
- [ ] 无 lint 错误
- [ ] 核心功能手动验证通过
```

---

## 任务粒度原则

- 每个 Task = 一个内聚功能单元（一个组件、一个 API、一个工具函数）
- 每个步骤 = 一个原子动作，2-5 分钟可完成
- 步骤顺序：**写测试 → 确认失败 → 写实现 → 确认通过 → 提交**
- 跨任务依赖必须在任务头部注明：`**依赖：** Task N 完成后才能开始`

---

## 质量要求

执行完步骤四后，过一遍 `references/quality-check.md` 清单，全部通过才输出。
