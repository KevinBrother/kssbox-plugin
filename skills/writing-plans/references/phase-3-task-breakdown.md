# 步骤三：拆解任务

## 目标

将文件结构转化为可执行的任务序列，每个任务交付一个可测试的功能单元。

## 任务块结构

每个任务必须包含：

```markdown
### Task N: [组件名]

> [这个任务交付什么，一句话。用户读完知道这个任务完成后系统能做什么]

**关联文件：**
- 新建：`path/to/file.ts`
- 修改：`path/to/existing.ts`

**依赖：** Task M 完成后才能开始（如无依赖则删除此行）

---

- [ ] 写失败的测试：[描述测试文件路径和测试意图]
- [ ] 运行测试，确认红色
- [ ] 实现最小代码让测试通过：[描述核心逻辑要点]
- [ ] 运行测试，确认绿色
- [ ] git commit -m "feat(scope): description"
```

## 任务粒度判断

| 信号 | 处理方式 |
|------|--------|
| 一个任务超过 8 个步骤 | 拆成两个任务 |
| 一个任务涉及 3 个以上文件 | 检查是否职责混淆，考虑拆分 |
| 任务之间没有依赖关系 | 记录为"可并行"，让执行者知道 |
| 有纯配置/脚手架任务 | 单独列为 Task 0（初始化），无需写测试 |

## 步骤描述要求

- **具体**：不写"实现功能"，要写"在 `useChat.ts:45` 新增 `abort()` 方法"
- **可验证**：每个任务必须有至少一个可运行的验证步骤
- **测试前置**：除 Task 0（纯脚手架）外，所有任务必须先写测试

## 提交信息格式

```
feat(scope): 描述          ← 新功能
fix(scope): 描述           ← bug 修复
refactor(scope): 描述      ← 重构
test(scope): 描述          ← 补充测试
chore(scope): 描述         ← 配置/依赖
```
