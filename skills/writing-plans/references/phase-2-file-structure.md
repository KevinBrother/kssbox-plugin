# 步骤二：设计文件结构

## 目标

在动手写代码前，明确所有文件的边界和职责，让后续的任务拆解有据可依。

## 执行动作

1. 读取步骤一提炼的功能模块列表
2. 对照已有代码库（如有），梳理哪些文件需要新建、修改、删除
3. 填写文件结构表格

## 文件结构表格格式

| 操作 | 路径 | 职责说明 |
|------|------|--------|
| 新建 | `src/components/ChatInput.tsx` | 输入框 UI，负责文本收集和发送触发 |
| 修改 | `src/hooks/useChat.ts` | 新增 abort 支持 |
| 删除 | `src/utils/legacy.ts` | 已废弃，本次清理 |

## 设计原则

- **单一职责**：每个文件只做一件事
- **按职能分组**：`components/`、`hooks/`、`services/`，而非按技术层分
- **就近原则**：经常一起改的文件放在一起
- **不擅自重构**：若存量代码有问题但不在本次范围，记录到计划末尾的"技术债"章节，不在本次计划里处理

## 完成标志

- 表格中每行都有明确的职责说明
- 没有"utils.ts"这类职责不清的命名
- 修改的文件标注了大致影响范围（如行号区间）
