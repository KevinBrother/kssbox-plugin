# 执行规则

## 核心原则

1. **计划是权威** — 不修改计划，只修改代码
2. **步骤不可跳过** — 每个 checkbox 都要实际执行，不能只是勾选
3. **测试必须运行** — 看到测试步骤，必须实际运行命令并等待输出
4. **遇阻立停** — 两次尝试无法解决的问题，停下来报告

## 每个任务的标准执行节奏

```
1. 宣告任务开始："开始 Task N: [组件名]"
2. 阅读该任务所有步骤（先通读，再执行）
3. 逐步执行，每步完成后打勾
4. 验证步骤：运行命令 → 等待输出 → 确认结果
5. 提交：git commit（message 与计划一致）
6. 宣告任务完成："Task N 完成 ✅"
```

## 测试执行规则

- 必须实际运行测试命令，不能跳过
- 如果测试框架未安装：记录到阻断报告，停止执行
- 如果测试失败：
  1. 第一次：检查实现，修复后重新运行
  2. 第二次：如仍失败，停止并报告（错误信息 + 已尝试的修复思路）

## Commit 规则

- 每个 Task 至少一个 commit，不合并多个 Task 的变更
- Message 格式必须与计划中标注的一致
- 如计划未标注 message，使用：`feat(<task-scope>): <task-description>`

## 范围控制

执行中**只做计划里有的事**，发现以下情况时：

| 情况 | 处理方式 |
|------|--------|
| 发现顺手能改的小 bug | 不改，记录到完成报告"建议后续处理" |
| 发现命名不规范的历史代码 | 不改，记录到完成报告 |
| 发现计划遗漏了某个文件 | 停止，报告给用户，确认是否补充计划 |
| 需要新增计划未提及的依赖包 | 停止，报告给用户，确认后再安装 |

## 完成报告格式

```
✅ 执行完成

完成任务：N / N
通过验收：N / N 条

建议后续处理：
- [发现的问题或改进点，不在本次范围内]

下一步：
- 若新增了模块/接口 → 运行 harness-sync 更新架构文档
- 若功能完整 → 可以发布/部署
```
