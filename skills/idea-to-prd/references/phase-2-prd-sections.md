## 第 2 章：用户故事

### 格式规范

每个用户故事的格式：

    US-[编号]
    作为 [用户角色]，
    我想要 [完成什么动作/看到什么]，
    以便 [达到什么目的/解决什么问题]。

    验收标准（AC）：
    - AC1：Given [前提]，When [操作]，Then [结果]
    - AC2：
    优先级：P0 / P1 / P2
    关联功能：F-XX

### 编写原则

- 每个 MVP 功能至少 1 个用户故事，重要功能写 2-3 个（主流程 + 边界情况）
- AC 必须可测试：测试人员能明确判断"通过/不通过"
- 不要在用户故事里描述实现方式
- 错误场景也要写（上传失败、网络断开、权限不足）

### 用户故事分组

按功能模块分组：

- 认证模块（注册、登录、找回密码）
- 核心功能模块 A
- 核心功能模块 B
- 权限与设置模块
- 支付模块（如有）

- BR1：[具体约束，如"免费用户最多创建 3 个项目"]

### 功能规格写作要点

- "系统应该"描述系统行为，"用户可以"描述用户操作

- 业务规则单独列，不要混入功能描述
- 边界情况必须覆盖：空状态、最大值、并发、网络失败、权限不足

### 5A：API 设计

**基础信息**

- Base URL：`/api/v1`
- 认证方式：[步骤一用户确认的方案，或填"待定，见第10章开放问题"]
- 内容类型：`application/json`

**统一响应格式**

成功响应：

```json
{
  "success": true,
  "data": {},
  "meta": { "page": 1, "total": 100 }
}
```

失败响应：

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "用户友好的错误描述"
  }
}
```

**错误码规范**

| 错误码 | HTTP 状态 | 含义 |
|------|---------|-----|
| UNAUTHORIZED | 401 | 未登录或 Token 失效 |
| FORBIDDEN | 403 | 无权限 |
| NOT_FOUND | 404 | 资源不存在 |
| VALIDATION_ERROR | 422 | 参数校验失败 |
| RATE_LIMITED | 429 | 请求频率超限 |
| INTERNAL_ERROR | 500 | 服务器内部错误 |

**端点格式（每个端点一节）**

格式示例（用文字描述，不用嵌套代码块）：

    [HTTP方法] [路径]
    描述：一句话说明用途
    认证：需要 / 不需要 / 可选
    权限：谁可以调用

    请求参数：
    | 参数名 | 位置 | 类型 | 必填 | 说明 |
    |------|-----|-----|-----|-----|
    | ...  | Path/Query/Body | String | ✅ | ... |

    请求体示例：
    { "field": "value", "other": 123 }

    响应示例（200）：
    { "success": true, "data": { "id": "uuid", "name": "..." } }

    错误情况：
    - 403：用户无权访问该资源
    - 422：字段校验失败，message 中包含具体字段名

**API 分组**：按模块组织（认证、用户、核心资源 A、核心资源 B、支付等）

## 第 9 章：开发优先级与里程碑

### 里程碑时长估算（动态，不要硬编码）

根据 MVP 功能数量和复杂度估算总工期：

| MVP 规模 | 参考工期 |
|--------|--------|
| 简单（≤5 个功能，复杂度低） | 4 周 |
| 中等（5-10 个功能，含第三方集成） | 8 周 |
| 复杂（>10 个功能，或含复杂业务逻辑） | 12 周以上 |

**在里程碑里说明估算依据**，例如：
> "本产品 MVP 共 7 个功能，含 Stripe 订阅集成，估算工期 8 周。"

### 功能优先级矩阵

| 功能 | 用户价值 | 开发复杂度 | 优先级 | 说明 |
|-----|--------|---------|------|-----|
| F-01 | 高 | 低 | P0 立刻做 | 核心流程必需 |
| F-02 | 高 | 高 | P0 尽早开始 | 工期长，提前启动 |
| F-03 | 中 | 低 | P1 | 增强体验 |
| F-04 | 低 | 高 | P2 推后 | ROI 低 |

### 里程碑计划

基于上方估算的总工期，划分阶段（示例为 8 周，实际按产品调整）：

**M1：基础搭建（Week 1-2）**

- 项目初始化、CI/CD 配置、数据库 Schema
- 认证模块（注册/登录）
- 基础 UI 框架搭建

**M2：核心功能（Week 3-5）**

- [F-XX] 核心功能 A
- [F-XX] 核心功能 B

**M3：集成与完善（Week 6-7）**

- [F-XX] 第三方集成（支付/通知等）
- 性能优化、错误处理完善

**M4：测试与上线（Week 8）**

- QA 全量测试
- Bug 修复
- 生产环境部署 + 监控接入
