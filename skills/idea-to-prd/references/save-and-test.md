# 落盘与自检流程

生成 PRD 内容后，必须执行以下两步，不可跳过。

---

## 步骤 A：落盘

将生成的 PRD 保存为 Markdown 文件。

**文件命名规则**：`prd-[产品名拼音或英文]-v1.0-[YYYYMMDD].md`
例：`prd-indie-crm-v1.0-20250413.md`

**保存路径**：当前项目根目录下的 `docs/` 目录（即 `./docs/`）。如果用户在其他目录中运行，佄倔询用户是否保存到当前目录。

**执行方式**：用 `create_file` 工具写入，内容为完整的 PRD Markdown 文档。

**写入完成后**：告知用户文件已保存到哪个路径。

---

## 步骤 B：自检

文件落盘后，重新读取刚写入的文件，对照以下清单**逐条**核查。每条都要判断"通过 ✅ / 不通过 ❌"，不通过的立即修复。

---

### B1 结构完整性

- [ ] 文档包含全部 11 个章节（第 0 章 → 第 10 章），无章节缺失
- [ ] 每个章节有实质内容，无 `[填写]`、`[待补充]`、`[F-XX]`、`$X/月` 等未替换占位符
- [ ] 第 10 章开放问题不为空（如确实无待决策项，写明"无，理由：……"）

---

### B2 用户故事

- [ ] 每个用户故事有编号（`US-XX` 格式），编号连续无跳号
- [ ] 每个用户故事有至少 2 条验收标准（AC）
- [ ] **AC 可测试性**：每条 AC 必须满足"测试人员能明确判断通过/不通过"
  - ❌ 不合格示例："系统应快速响应"、"用户体验良好"
  - ✅ 合格示例："Given 用户未登录，When 访问 /dashboard，Then 系统跳转到 /login 并保留原始 URL 参数"
- [ ] 至少有 1 个用户故事覆盖错误场景（登录失败、网络断开、权限不足等）
- [ ] 所有 MVP 功能都有对应的用户故事（无功能被遗漏）

---

### B3 功能规格

- [ ] 每个 MVP 功能有 `F-XX` 编号，编号连续
- [ ] 功能规格只描述"系统做什么 / 用户能做什么"，不包含实现技术（如"用 React"、"调用 Redis"）
- [ ] 每个功能的**边界情况（BC）**至少覆盖：
  - 空状态（无数据时的行为）
  - 最大值/限制触发（如超出免费配额）
  - 权限不足时的响应
- [ ] 每个功能的**业务规则（BR）**有具体数字或条件，不用"适量"、"若干"、"较多"
- [ ] `关联用户故事` 字段填写了真实存在的 US 编号（无断链）

---

### B4 数据模型

- [ ] 从功能规格反推，所有被操作的数据对象都有对应实体（无遗漏）
- [ ] 每个实体至少包含：`id`（UUID）、`created_at`、`updated_at`
- [ ] 所有外键字段已在索引列表中
- [ ] 实体关系（一对多 / 多对多）有明确文字描述
- [ ] 枚举字段列出了所有合法值（不能只写"Enum 类型"）

---

### B5 API 设计 / 状态管理

- [ ] 每个 MVP 功能对应至少一个 API 端点或状态变更定义
- [ ] **API 端点**：
  - 请求体有具体字段名和类型，不是空的 `{}`
  - 响应体有具体字段示例，不是 `{ "data": {} }`
  - 列出了至少 2 种错误情况（含触发条件）
- [ ] **状态管理**（纯前端产品）：
  - 每个状态有类型、初始值、持久化方式
  - 关键状态变更有触发条件和副作用描述

---

### B6 页面与交互流程

- [ ] 页面清单覆盖了所有 MVP 功能涉及的页面（无功能有 API 但无对应页面）
- [ ] 每个核心用户流程都写了**正常流**和**异常流**
- [ ] **异常流**必须覆盖：
  - 表单校验失败 → 具体提示方式（字段级 / 顶部 Toast）
  - 网络请求失败 → 用户看到什么，能做什么操作
  - 权限不足 → 跳转逻辑或提示内容
- [ ] 关键页面规格包含三种状态描述：**空状态 / 加载中 / 错误状态**
- [ ] 页面间跳转逻辑完整（进入方式 + 离开方式都有）

---

### B7 非功能性要求

- [ ] 性能指标有具体数字（FCP 秒数、API P95 毫秒数、并发用户数）
- [ ] 安全要求说明了认证方案、Token 有效期、密码哈希算法
- [ ] 合规要求明确勾选了适用项（GDPR / 数据本地化 / 其他），不是全部留空

---

### B8 验收标准

- [ ] 功能验收表的每行都引用了具体 AC（不是"参见用户故事"这种泛指）
- [ ] 关键指标验收有具体数字目标（如"核心流程成功率 > 99%"）
- [ ] 上线 Checklist 全部条目保留，无删减

### B8.5 Harness Engineering 验收

- [ ] 第 7.6 章存在且完整（AGENTS.md 要求 / 分层架构 / CI 检查清单 / 人工节点）
- [ ] CI 强制检查列出了：类型检查、linter、模块边界、覆盖率四项，每项说明了工具和失败行为
- [ ] 分层架构依赖方向明确（Types → Config → Repository → Service → Runtime → UI）
- [ ] 里程碑 M1 中包含 AGENTS.md 和 docs/architecture.md 的编写任务
- [ ] 每个里程碑后有架构一致性检查时间（哪怕只写"0.5天 pattern drift 检查"）
- [ ] 人工节点（human checkpoint）在里程碑中有明确标注，不是全程 agent 自主

---

### B9 里程碑

- [ ] 总工期和 MVP 功能数量匹配：≤5功能→4周 / 5-10功能→8周 / >10功能→12周以上
- [ ] 文档中写明了估算依据（功能数 + 特殊复杂度说明）
- [ ] 功能依赖关系已列出（认证功能必须在其他功能之前）

---

### B10 逻辑一致性（交叉核查）

- [ ] US 编号 ↔ F 编号：每个 F-XX 的"关联用户故事"字段填的 US 编号真实存在
- [ ] F 编号 ↔ 数据实体：功能规格里提到的所有数据对象在第 4 章有对应实体
- [ ] F 编号 ↔ API 端点：每个 MVP 功能在第 5 章有对应端点（或状态变更）
- [ ] F 编号 ↔ 页面：每个需要 UI 的功能在第 6 章有对应页面
- [ ] 业务规则数字一致：免费版限制数字在功能规格（BR）、验收标准（AC）、开放问题中保持一致，无矛盾

---

### 发现问题时的处理规则

| 问题类型 | 示例 | 处理方式 |
|--------|-----|--------|
| 轻微（格式/占位符） | `[F-XX]` 未替换、编号跳号 | 直接修复，重新保存，**不通知用户** |
| 实质性（内容缺失/逻辑错误） | AC 不可测试、实体缺失、异常流为空 | 修复后**告知用户改了什么** |
| 无法自动修复（信息不足） | 输入文档未说明并发要求 | 放入第 10 章开放问题，**在报告中标注** |
