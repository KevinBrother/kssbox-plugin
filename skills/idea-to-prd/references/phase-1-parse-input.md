# 步骤一：解析输入文档

从用户提供的产品规划文档中提取以下信息。**提取完成后不要直接进入生成，先向用户确认。**

## 提取清单

### 基本信息
- [ ] 产品名称
- [ ] 产品类型（Web/App/插件/API/桌面）
- [ ] 目标平台（iOS/Android/Web/Desktop/跨平台）
- [ ] 技术栈偏好（如有）
- [ ] 是否有后端/API？（决定第 5 章写 API 设计还是状态管理规格）

### 用户信息
- [ ] 主要用户角色（1-3 个，用于用户故事的"作为 [用户]"）
- [ ] 每个角色的核心场景

### 功能信息
- [ ] MVP 功能列表（优先级排好序）
- [ ] v1.1 功能列表
- [ ] 明确不做的功能

### 业务规则
- [ ] 免费/付费限制（具体数量/功能边界）
- [ ] 用户权限规则（角色/团队/权限）
- [ ] 数据保留/删除规则（如有）

### 技术约束
- [ ] 第三方集成（支付/邮件/登录/存储等）
- [ ] 性能要求
- [ ] 安全要求
- [ ] 合规要求（GDPR/数据本地化等）

### 验收指标
- [ ] 激活指标
- [ ] 留存指标
- [ ] 转化指标
- [ ] 性能指标

## 提取原则

- **有冲突时**：以第十一章"PRD 输入摘要"为准，其他章节为背景补充
- **信息缺失时**：记录下来，进入步骤二后放入"开放问题"章节，不要自行假设
- **功能描述模糊时**：拆解成最小可描述的子功能，再逐一写规格

## 提取完成后：向用户确认

提取完成后，**不要直接开始生成 PRD**。先用以下格式向用户确认关键信息：

> "我已经读取了你的规划文档，准备生成 PRD。确认一下关键信息：
> - 产品：[产品名]，类型：[Web/App/...]，[有/无]独立后端
> - MVP 功能：[功能1]、[功能2]、[功能3]...（共 X 个）
> - 技术栈：[已确定：... / 待定]
> - 发现 [X] 个信息缺口，将在开放问题章节列出
>
> 确认无误我就开始生成，或者告诉我需要调整的地方。"

**等用户明确确认后**，再进入步骤二。
