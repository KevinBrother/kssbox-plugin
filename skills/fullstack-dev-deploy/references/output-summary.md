# 输出摘要

## 目标

在 skill 执行完成时，输出一段结构化摘要，让用户快速知道识别结果、治理动作、最终保留资产和验证情况。

## 必须包含的字段

- 识别到的 app 列表
- `all` 映射了哪些 app
- `biz` 映射了哪些 app
- `boss` 映射了哪些 app
- 哪些旧资产被 keep / migrate / merge / delete / generate
- 最终保留了哪些 dev / deploy / docker / env 入口
- 新建了哪些文件
- 修改了哪些文件
- 删除了哪些文件
- 哪些地方按假设处理
- 哪些验证已执行

## 推荐输出顺序

1. 项目画像摘要
2. 服务分组摘要
3. 治理动作摘要
4. 最终资产摘要
5. 文件变更摘要
6. 假设项摘要
7. 验证摘要

## 项目画像摘要

至少说明：

- 识别到多少个 frontend app
- 识别到多少个 backend app
- 每类 app 的主要技术栈

## 治理动作摘要

至少区分：

- 哪些旧脚本被 keep
- 哪些旧脚本或配置被 migrate
- 哪些资产被 merge
- 哪些旧文件被 delete
- 哪些标准资产是本次 generate 出来的

## 最终资产摘要

至少说明：

- 最终是否保留 `scripts/dev.sh`
- 最终是否保留 `scripts/deploy.sh`
- 是否形成 `docker/infra.compose.yml`
- 是否形成 `docker/app.compose.yml`
- 哪些 app 拥有自己的 `.env.dev.example` / `.env.prod.example` / `.env.local.example`

## 文件变更摘要

至少区分：

- 新建了哪些文件
- 修改了哪些文件
- 删除了哪些文件
- 哪些旧资产被收敛到新入口

## 假设项摘要

如果存在自动推断，必须显式说明：

- 哪个映射、命令或归属是按假设处理
- 为什么能继续执行
- 用户后续如果要调整，应改哪一层配置

## 验证摘要

必须说明：

- 哪些治理校验已执行
- 哪些 dev 验证已执行
- 哪些 deploy 验证已执行
- 哪些验证通过
- 若有未执行的验证，是因为什么前置条件缺失

## 禁止行为

- 只说“已完成”，不列出具体治理动作和文件变化
- 不提假设项，导致用户误以为所有判断都已确认
- 不提最终保留资产，让用户不知道仓库最终长什么样
- 不提验证范围，让用户不知道是否真的跑过
