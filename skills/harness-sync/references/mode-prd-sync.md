# 模式：PRD-SYNC PRD 变更反哺

PRD 更新后，将架构决策变化同步回 Harness 文件。

## 核心原则

PRD 是需求和架构决策的上游。PRD 改了，Harness 必须跟着改——
否则 agent 拿着过期的 AGENTS.md 写代码，写出来的东西和新 PRD 矛盾。

## 需要用户提供

- 新版 PRD（或变更的章节）
- 旧版 PRD（或描述"之前是什么，现在改成什么"）

如果只有新 PRD，问用户：**"这次 PRD 主要改了什么？是需求调整、架构决策变更、还是两者都有？"**

## 执行流程

### 第一步：提取 PRD 变更点

不依赖固定章节编号——扫描文档中**信息类型**的变化，而不是找特定章节：

| 信息类型 | 对应 Harness 影响 |
|--------|----------------|
| 架构约束 / Harness 要求（分层规则、CI 门禁、覆盖率基线） | 直接影响所有 Harness 文件 |
| 数据模型 / 领域实体（新增实体、字段变化、关系变化） | architecture.md 的模块层级 |
| API 设计 / 接口契约（新增端点、请求响应结构变化） | architecture.md、AGENTS.md |
| 功能规格 / 业务规则（新增限制条件、权限策略、计算逻辑） | AGENTS.md 禁止事项（新业务规则 → 新约束） |
| 里程碑 / 交付节点（MVP 范围调整、阶段划分变化） | AGENTS.md 的人工节点说明 |
| 未决问题 / 开放决策（标注"待定"的问题已落地决策） | 已决策的问题 → 新增 ADR |

### 第二步：分类变更

**架构决策变更**（需要新增 ADR）：

- 技术栈变化（如从 REST 改 GraphQL）
- 新增重要第三方集成
- 分层架构调整
- 数据存储方案变化

**约束变更**（更新现有 Harness 文件）：

- 业务规则调整 → 更新 AGENTS.md 禁止事项
- 模块边界变化 → 更新 architecture.md + dependency-cruiser
- 新的命名规范 → 更新 conventions.md

**里程碑/人工节点变化**（更新 ADR 或 AGENTS.md 的不确定情况节）

### 第三步：生成变更

对每个受影响的文件，生成具体的更新内容（diff 格式，见 mode-update.md）。

新增 ADR 时，格式：

```markdown
# ADR-[NNN]: [决策主题]

**日期**：[日期]
**状态**：已确认
**触发**：PRD v[旧版本] → v[新版本] 变更

## 背景
[PRD 为什么要做这个变更，原来的方案有什么问题]

## 决策
[新的架构决策是什么]

## 对现有代码的影响
- [需要重构的部分]
- [不受影响的部分]

## Harness 联动更新
- [同步更新了哪些 Harness 文件，改了什么]
```

### 第四步：输出变更摘要 + 影响评估

```
## PRD 变更 → Harness 同步报告

**PRD 变更摘要**：[一句话描述 PRD 改了什么]

**对现有代码的影响评估**：
- 🔴 高影响：[需要重构的模块，估计工作量]
- 🟡 中影响：[需要调整但不需要重构的部分]
- 🟢 低影响：[只需更新文档，代码不用动]

**Harness 更新清单**：
| 文件 | 操作 | 原因 |
|-----|-----|-----|
| ... | ... | ... |

**建议下一步**：
1. [先做什么，再做什么]
2. [需要人工决策的开放问题]
```
